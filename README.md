# Hybrid Fatawa Retrieval

A multilingual Retrieval-Augmented Generation (RAG) system over ~170k Urdu
Islamic fatawa from four schools of thought (Banuri / Deobandi, IslamQA and
FatwaQA / Ahle Hadees, UrduFatwa / Barelvi). Three retrieval modes are
exposed through one Flask UI:

1. **Hybrid** — dense (Pinecone, `text-embedding-3-small`) + sparse (BM25 via
   `bm25s`) + LLM synthesis (`gpt-4o-mini`).
2. **PageIndex** (vectorless) — LLM navigation over a markdown hierarchy
   (H1–H6) of the corpus.
3. **Raw Fatwas** — tree-guided keyword search over an inverted index with
   LLM re-ranking.

Queries can be written in **Urdu, English, or Roman Urdu** (formal or
informal); answers are always returned in formal Urdu. A single LLM
preprocessor handles language detection, translation, and fiqh-synonym
expansion.

## Requirements

- Python 3.11 (a `.venv311` virtualenv sits at the workspace root)
- `OPENAI_API_KEY` and `PINECONE_API_KEY` in `.env` (see
  [src/config/settings.py](src/config/settings.py) for all keys)
- ~2 GB RAM for the in-memory BM25 index
- Pinecone serverless index `fatawa-hybrid` (us-east-1, cosine, 1536-dim)

## Quick start

```bash
pip install -r requirements.txt
cp .env.example .env   # fill in keys

# Dry-run mode (no API keys, mock backend):
python app.py --dry-run

# Production:
python app.py                    # http://127.0.0.1:5000
python app.py --port 8080        # alt port
python app.py --no-guardrails    # disable safety guards
```

## Ingestion

The ingest pipeline has 6 stages. Both entry points expose the same stages;
each stage is independently re-runnable from `orchestrator.py`'s
`run_stage_N` functions.

```bash
# Preferred — auto-resumes via pipeline_state.json + embed_checkpoint.db
python orchestrator.py                                  # full run
python orchestrator.py --force                          # re-run everything
python orchestrator.py --stages 5,6                     # embed + upsert only
python orchestrator.py --dry-run                        # no API calls

# Background, logs to pipeline_qonly.log:
scripts/resume_qonly.sh
scripts/status_qonly.sh
```

## Running queries

```bash
python run_query.py                              # REPL
python run_query.py "آپ کا سوال یہاں"             # one-shot
python run_pipeline.py --mode query --guardrails --validate --question "..."
```

## Architecture notes

- **Multilingual preprocessor**: `_preprocess_query()` in [app.py](app.py)
  calls `gpt-4o-mini` once to detect language, translate to formal Urdu, and
  emit fiqh synonyms. The retrieval query is then built deterministically as
  `urdu_question + " (" + synonyms + ")"` so that formal-Urdu and
  Roman-Urdu versions of the same question produce identical lexical
  anchors for BM25.
- **BM25**: `bm25s` library, numpy-vectorised — ~1–5 ms per query over
  167k docs. Cache at `.bm25_cache.pkl`, rebuilt from the embedding
  checkpoint (`BM25Corpus.build_from_checkpoint`) on cache miss.
- **Schema-driven ingest**: the four CSV sources have heterogeneous
  columns. `src/analysis/schema_mapper.py` infers per-file mappings into
  a unified record (`id`, `question`, `answer`, `text`, `category`, …)
  consumed by every downstream stage.
- **Guardrails**: `src/pipeline/guardrails.py` runs pre- and post-LLM
  checks (context score, overlap, Urdu output ratio, length). The
  `NO_ANSWER_SENTINEL` string in `prompt_builder.py` is special-cased by
  both guards and the output validator — preserve it exactly when editing
  prompts.
- **Entry-point ordering**: every entry point (`app.py`, `run_pipeline.py`,
  `orchestrator.py`, the tests) parses `--dry-run` and installs
  `DryRunContext` patches **before** importing `src.*`. Preserve this
  ordering when editing — `src` imports will bind to the real OpenAI /
  Pinecone clients on import and can't be swapped afterwards.
